The problem we solve
flowchart LR
  A[UI Changes] --> B[Broken Selectors]
  B --> C[Test Failures]
  C --> D[Maintenance Cost]
  D --> E[Slow Feedback]
Typical automation pain: fragility + time spent fixing locators.
  • Flaky runs from timing and dynamic DOM states
  • Selector churn during UI changes
  • Repeated boilerplate (waits/locators) across tests
  • Limited traceability on what changed and why
Solution Centralize waits + encapsulate pages + add optional AI healing with audit trail.
Architecture overview
flowchart TB
  T[TestNG Tests] --> F[SeleniumFactory]
  T --> P[Page Objects]
  P --> B[BasePage (waits + smart actions)]
  P --> E[ElementInitializer (reflection)]
  E --> S[SmartElement (By + driver)]
  B -->|Timeout| A[AIHealingEngine]
  A --> L[AI audit log]
Simple flow: Tests call Pages; Pages rely on Core; Core optionally triggers AI healing.
Clean layering improves readability and ownership.
Core implementation (highlights)
  • `SeleniumFactory`: singleton WebDriver + config (`browser.type`, `browser.headless`)
  • `BasePage`: navigation + explicit waits + `smartClick/smartFill`
  • `@SeleniumSelector`: clean element declarations
  • `ElementInitializer`: reflection wiring → `SmartElement(By)`
Selector syntax: CSS (default) or XPath with xpath=...

Smart Actions: stable by default
flowchart TB
  A[smartClick / smartFill] --> B[Parse selector to By]
  B --> C[Explicit wait]
  C -->|Success| D[Act (click/sendKeys)]
  C -->|Timeout| E[AI Healing attempt]
Centralized waits reduce flakiness and keep tests readable.
  • Explicit waits only (implicit wait set to 0)
  • One retry policy to avoid hiding real failures
  • Stale element retry once (common transient Selenium issue)
  • Audit friendly logs for healing decisions

AI hardening (real-world guardrails)
flowchart LR
  A[Timeout] --> B[Cache?]
  B -->|Hit| C[Validate selector]
  B -->|Miss| D[LLM heal]
  D --> C
  C -->|Valid| E[Retry once]
  C -->|Invalid| F[Bypass cache once + fresh heal]
Resilience without chaos: cache + validation + single retry.
  • Redaction: best‑effort masking before prompts/audit logs
  • Persistent cache: `healing-cache.json` stores successful heals
  • Validation: accept only selectors that match exactly one element and are actionable
  • Cache bypass: if cached heal is stale, bypass once and ask again
Cache path override: mvn test "-Dhealing.cache.path=...json"
